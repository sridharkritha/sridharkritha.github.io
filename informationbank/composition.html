Composition



The 2 most common techniques for reusing functionality in Object-oriented systems are Class Inheritance and Object Composition.

    1. Class Inheritance (Is-a / Kind of): class D is derived from the class B
    2. Object Composition(has-a)/Aggregation: A class can contain objects of other classes as its member. i.e., the Object of one class is contained in
        another class as a data member.
        Ex: 1. We might place an Engine object inside a Car object as member data because a "car has an engine"
              2. Timer Class has hours object and minute object

Note: 1. If you don't want to override the base class function in the derived class through inheritance then use composition instead.
          2. Composition Example: Using vector, set, map object in a user-defined class.

Delegation: Composition is about the relationships between objects. Delegation is about passing work from one object to another. These are actually different but they are related. Delegation is nothing more than an implementation detail.

Use of Composition: If you have a ClassB that you'd like to have inherited from both ClassA and ClassC, instead you can use the composition for ClassA or ClassC. Figure 4.8 shows how this looks - 
    1. ClassB is inherited from ClassA and ClassC (is-a)
    2. ClassB composed of ClassC(has-a).

Use composition to avoid multiple inheritances

Note: A good software design always avoids multiple inheritances by composition. If you use multiple inheritances then the possibilities for missing the memory deallocation for the inherited classes is more that's the main reason most of the high-level languages like C#, Java not supporting multiple inheritances and supporting automatic garbage collection which C++ does not support.

    class remote {
    public:
        int* r;
        remote() {
            r = new int[3];
            r[0] = 5;
            r[1] = 6;
            r[2] = 7;
        };
        ~remote() { delete r; };
    };

    class base {
    public:
        int* p;
        remote* r_; // wil not call remote constructor (need to call explicitly using new operator)
        remote rr_; // will call remote constructor automatically
        base() {
            p = new int[3];
            p[0] = 5;
            p[1] = 6;
            p[2] = 7;
        }
        ~base() { delete p; }
        int* getValue() { return p; }
    };

    class composition {
    public:
        base b_;
        composition() { }
        ~composition() { }
    };

    class derived: public base {
    public:
        int* q;
        derived() { 
            q = new int[3];
            q[0] = 15;
            q[1] = 16;
            q[2] = 17;
        }
        ~derived() { delete q; }
    };

    int main()
    {
        base b; // new memory allocated for "p"
        cout<< "Addr of p = " << b.getValue() <<endl;

        composition c; // once again new memory is allocated for "p" bcos of composition
        cout<< "Addr of p = " << c.b_.getValue() <<endl; // address of p is different from base allocated address of p.

        derived d; // once again new memory is allocated for "p" bcos of inheritance
        cout<< "Addr of p = " << d.getValue() <<endl; // address of p is different from base allocated address of p.

        derived* dd = new derived();
    }

Note: If a base class constructor allocating memory inside the class then any class which derive from the base class / compositing the base class
will once again call the base constructor to create another new memory again in the base class. It is not possible to reuse the already created
memory in the base class ??

Solution: Instead of creating an object for the base class, create an object for the most derived class. By doing this we can have only one object with
a unique memory for all the class. Ex: create an object for either derived class or composition class not for the base class
