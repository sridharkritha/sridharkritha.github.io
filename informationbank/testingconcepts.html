Testing Concepts
 
 Quality Assurance

  1. Software Engineering:

It is the technology and managerial discipline concerned with systematic production and maintenance of software products.

 

Testing:

Executing the software with data to ensure that the software works correctly.

  1. Integration testing: Testing during the integration of the software.
  2. System testing: Testing the s/w in an environment that matches the operational environment.
  3. Accepting Testing: testing to satisfy the purchaser.
  4. Unit testing: Testing by the original developer.
  5. Alpha testing: testing by the customers at the developer’s site.
  6. Beta testing: Testing by the customers at the customer site.
  7. Regression testing: Saving tests form the previous version to ensure that the new version retains the previous capabilities.

Software Cost estimation factors:

  1. Programmers’ ability
  2. Product complexity
  3. Product size
  4. Available time
  5. Level of technology
  6. Required reliability.

 Software cost estimation technique:

  1. Expert judgment:

It is a “top down estimation” technique, here expert judgment relies on the experience, background and business sense of one or more key people in the organization.

 

  1. Delphi cost estimation:
    1. Here a coordinator provides a “form” for recording a cost estimate, to each estimator [estimation can be done based on system definition given by the coordinator]
    2. Estimators study the definition and complete their estimation.
    3. Estimators have any doubt, they can ask questions to the coordinator but they do not discuss their estimates with one another.
    4. Coordinator collects the estimations and re-distributes a summary of estimators response and includes any unusual rationales noted bu the estimator.
    5. Estimators again compute the estimation on the basis of previously estimated summary of estimations.
    6. Estimators whose estimates differ sharply from the group should provide the clear justification for their estimates.
    7. The process is iterated for as many of rounds as required.
    8. But “no group discussion” is allowed during the entire process.

 

Work Break sown Structure:

It is a “bottom –up” estimation tool. Work break down structure is a hierarchical chart that accounts for the individual parts of the system. A “WBS” can use “product hierarchy” and “process hierarchy” for cost estimation.

A. Linear sequential model / Waterfall model / phased life cycle model

 

This model segments the S/W life cycle in to a series of successive activities. 

 

Feasibility Test phase

Requirement Phase

Design phase

[Architectural details]

Implementations Phase

[Coding/ Debug / Unit Test]

System Test

[Integration / Acceptance Test]

Maintenance Phase

[Enhancement]

 

 

B. Prototype Life cycle model:

Prototype is a “mock-up” or “model” of s/w products. Prototype is used to demonstrate the proposed behavior input data format and interactive features to the customers. After agreement from the customer, then the software development will start as like waterfall method.

C. Incremental Model / Successive model:

This is an extensive of prototype model. The goal of the model was to design and delivery a minimal subset of the whole system to the customers, that was still a useful sub system. The processes will continue to iterate through the whole life cycle with additional minimal increment.

 

Mile stone:

Mile stones are events that can be used for telling the status of the project. “ Two” major milestone during software design are

 

    1. Preliminary design review [PDR]: held near the end of “architectural design” and prior to detailed design.
    2. Critical design review [CDR]: held near the end of “ detailed design” and prior to implementation.

Walkthrough :

  • A walkthrough is not a project review but is a in-depth, technical review of some aspect of a software system.
  • Walkthrough can be used at any time, during any phase of software projects
  • Walkthrough team usually consists of “reviewee” and 3-4 reviewers
  • Walkthrough team may include project leader, other member of the project team, a representative from the quality assurance group, a technical writer excepts higher level managers.
  • In walkthrough session, the project being examined is presented by a “reviewee” and evaluated by team of reviewers [reviewer look for error, request clarifications from the “reviewee”]
  • Walkthrough session should not be used to correct error, only for finding the error.
  • Walkthrough session should be limited to,” 2hours”.

Inspection:

  • Inspection team consist of “1-4” members who are trained for their task
  • The team consists of 1. Moderator, 2. Designer 3. Implementer and 4. Tester.
  • There are “3” separate inspections are done on a project 1. Design inspection 2. Implementation inspection and 3. Unit testing.

 

Four Categories of test that a programmer will typically perform on program unit :

 

Black Box Test:

    1. Functional Test: Excise the function code with nominal I/P values for which the expected result are known.
    2. Performance test: Amount of execution time spends in various parts of the unit.
    3. Stress Test: To find strength and limitations of a program.

 

White Box Test:

                   4. Structure Test: Exercising the internal logic of a program & traversing executional path of the program.
