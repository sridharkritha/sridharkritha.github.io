GDB Miscellaneous
 
 

Core Dump files:

Running programs under gdb makes for easier bug trapping, but usually a program will die outside of the debugger leaving only a core file. Gdb can load core files and let you examine the state of the program before it died.

 

For Example: divide.cpp

   #include <stdio.h>   

   int main(void)

   {

            int div,a,b;

            printf("Inside main - Start\n");

            a = 5;

            b = 0;

            div = a/b; /* Divide by Zero */

            return 0;

   }

 

Procedure:

 

% g++ -g divide.cpp -o sri

% ./sri

 

Output: Floating point exception (core dumped)

 

Solution:

 

% gdb sri core

% bt

 

Attaching GDB to running processes:

 

In addition to debugging with programs or core files, gdb can attach to an already running process (has debugging information). This is done by specifying the process ID of the program you wish to attach gdb to instead of the core filename.

 

Here's an example program which goes round in a loop and sleeps: Loop.cpp:

 

#include

int main(int argc, char *argv[])

{

  int i;

  for(i = 0; i < 60; i++)

  {

    sleep(1);

  }

  return 0;

}

 

Solution:

 

% g++ -g loop.cpp -o sri

% ./sri &

 

Output: [3] 1256

 Take note of the process ID which is printed when it starts in the background, in this case 1256.

 

% gdb sri 1256

% bt

 

More Options:

Command

Description

b function-name
b line-number

b filename:function

b filename:line-number

b line-number if condition

Break program at specified function or line number

d

Delete all breakpoints, watchpoints, or catch points.

d bp-num
d range

Delete the breakpoints, watchpoints, or catch points of the breakpoint ranges specified as arguments.

Ex: d 2-9

disable bp-num or range

Does not delete breakpoints. Just disables them.

enable bp-num or range

Does not delete breakpoints. Just enables them.

continue number

Continue but ignore current breakpoint number times. Useful for breakpoints within a loop.

finish

Continue to end of function.

s no.ofSteps-to-perform

Step to next line of code. Will step into a function.

f number

Select frame number.

p variable-name
p file-name::variable-name

Print value stored in variable.

p *array-variable@length

Print first # values of array specified by length. Good for pointers to dynamically allocated memory.

p/x variable

Print as integer variable in hex.

p/t variable

Print as integer value in binary.

i local

Info aboutt all the variable values in the fun.

i break

Info about the breakpoints:

i args

Shows program command line arguments

i threads

Shows threads in use

i frame

Shows address, language, address of arguments/local variables and which registers were saved in frame.

list
l
l line-number
l function

List source code.

r command-line-arguments

Run the program with arguments

q

Exit GDB debugger.
