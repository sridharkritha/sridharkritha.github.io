1. How to add user method to the build-in String class ?
Method 1:
If you want to modify the prototype of a built-in, it's best (by far) to make that a non-enumerable property by using Object.defineProperty.
    Object.defineProperty(String.prototype, "SayHi", {
        value: function SayHi() {
            return "Hi " + this + "!";
        },
        writable: true,
        configurable: true
    });
    // The default for enumerable is false.

Method 2: (Not recommended)
    // Don't do this if you can use `Object.defineProperty`
    String.prototype.SayHi = function SayHi() { return "Hi " + this + "!"; };
Never do that with Array.prototype or Object.prototype; creating enumerable properties on those is a Bad Thing.

2. How to apply string operation on general Object (NOT a String Object) ?
    { a: 'hello', b: 'world' }.split('') =>  [ 'h', 'e', 'l', 'l','o', ' ', 'w', 'o','r', 'l', 'd']

Method 1:
    function SomeObject(a, b) { this.a = a; this.b = b; }
    // converts the object to string
    SomeObject.prototype.toString = function() { return this.a + " " + this.b; };

    var obj = new SomeObject('hello', 'world');
    console.log(obj); // { a: 'hello', b: 'world' }
    // console.log(obj.split("")); // TypeError: obj.split is not a function
    console.log('' + obj); // hello world   <- internally calls toString for the string conversion
    console.log(('' + obj).split("")); //   [ 'h', 'e', 'l', 'l','o', ' ', 'w', 'o','r', 'l', 'd']

Method 2: Inherit from String Class: You can let your objects inherit from String so that all string methods become available:
All the String.prototype methods (except for .toString, .valueOf and [Symbol.iterator]) are "intentionally generic; they do not require that its this value be a String object. 
Therefore, they can be transferred to other kinds of objects for use as a method." You can call them on any value, they will coerce it to a string (using .toString() or .valueOf as usual).
    function SomeObject(a, b) { this.a = a; this.b = b; }
    // Inherit from the String Class
    SomeObject.prototype = Object.create(String.prototype);
    SomeObject.prototype.constructor = SomeObject;
    // converts the object to string
    SomeObject.prototype.toString = function() { return this.a + " " + this.b; };

    var obj = new SomeObject('hello', 'world');
    console.log(obj);           // { a: 'hello', b: 'world' }
    // console.log('' + obj);   // TypeError: Bcos of String inheritance String.prototype.valueOf 
                                // requires that 'this' be a String
    console.log(obj.split("")); // [ 'h', 'e', 'l', 'l','o', ' ', 'w', 'o','r', 'l', 'd']

Inheritance: ES6 Vs ES5
Advantages of using ES6:
    1. ES6 uses the same prototypical inheritance we've always had, just with cleaner and more convenient syntax if you like using constructor functions (new Foo).
       (Particularly in the case of deriving from Array or Error, which you couldn't do in ES5 and earlier. )
    2. If you don't use constructor functions in the first place, preferring Object.create over class.

If you do use constructor functions, there are some benefits to class:
    a. The syntax is simpler and less error-prone and easy to set up inheritance hierarchies using the new syntax than with the old.
    b. class defends you from the common error of failing to use new with the constructor function (by having the constructor throw an exception
        if this isn't a valid object for the constructor).
   c. Calling the parent prototype's version of a method is much simpler with the new syntax than the old (super.method() instead of 
      ParentConstructor.prototype.method.call(this) or Object.getPrototypeOf(Object.getPrototypeOf(this)).method.call(this)).

Constructor Return Value: The constructor returns the 'this' object.
By the Javascript spec, when any function is invoked with new, Javascript creates a new object / {}, then sets the "constructor" property of that object to the function invoked, and finally assigns that object to the name 'this'. You can access that object using 'this' from inside the body of the function.
That's the object you have access with the this keyword inside the constructor when called with the new keyword.

Example: Default return type of Constructor
    function Car(){
        this.num_wheels = 4;
        // return { num_wheels:37 }; // NO need to return, JS will do it for you
    }
    var car = new Car();   // car = { num_wheels:37 }

Example: Explicitly returning non-object(primitive) type from the Constructor is ignored
    function Car() {
        this.num_wheels = 4;
        return 'VROOM'; // Returning Non-Object type is ignored
    }

    var car = new Car();   // car = { num_wheels:37 }
    alert(car.num_wheels); // 4
    alert(Car());          // 'VROOM' bcos without 'new'

Example: Explicitly returning object type from the constructor
    function Car() {
        this.num_wheels = 4; // the object referred by `this` will be lost...
        return {"name": "YamahaBike"}; // Returning Object type will Override the 'this' object
    }

    var car = new Car();         // car = { name: 'YamahaBike' }
    console.log(car.num_wheels); // undefined
    console.log(Car());          // { name: 'YamahaBike' }

Example: Explicitly returning object type from the constructor
    function Car() {

        return {"name": "YamahaBike", num_wheels : 2}; // Returning Object type will Override the 'this' object

Object Literal Vs Constructor Function: 
Object literals are basically singletons with variables/methods are all public.
Constructor function lets you to create the multiple instances of that class/constructor function. 

 Purpose Literal Constructor
 Global Instance  YES NO
 Multiple Instances NO YES
 OOP Feature  NO YES

    // Create an Object by Object literal
    var objLit = {
        x: 0, y: 0, // everything is public
        add: function () { return this.x + this.y; } // Note: objLit.x and objLit.y can be access outside
    };

    // Define a constructor function for creating the object(s) out of it by the 'new'
    var ObjCon = function(_x, _y) {
        var x = _x; // private
        this.y = _y; // public
        this.add = function () {
            return x + this.y; // x doesn't need 'this'
        };
    };

    // use the objects:
    objLit.x = 3; 
    objLit.y = 2; 
    console.log(objLit.add()); // 5

    // Multiple independent copies of objects can be created using the same constructor
    var objConIntance = new ObjCon(5,4);  // instantiate an objCon
    console.log(objConIntance.add()); // 9
    console.log((new ObjCon(7,8)).add()); // 15 - another instance of objCon
    console.log(objConIntance.add());  // 9 same result, not affected by the previous line

Inheritance in short:   Every object has an internal property, [[Prototype]], linking it to another object or null:
    object [[Prototype]] → anotherObject

In traditional javascript, the linked object is the prototype property of a function:
    object [[Prototype]] → aFunction.prototype

Some environments expose [[Prototype]] as __proto__:
    anObject.__proto__ === anotherObject

There are 5 ways you can read/write/create the [[Prototype]] link when creating an object.
    // (1) new operator (configures the prototype chain on the default object returned from a constructor function),:
    var object = new aFunction;
    // object.__proto__ = aFunction.prototype

    // (2) Object.create will set the supplied argument as the [[Prototype]] of the resulting object,
    var object = Object.create(anotherObject)
    // object.__proto__ = anotherObject

    // (3). The extends keyword (configures the prototype chain when using the 'class' syntax)

    // (4) Standardized(ES6) accessor (ie. getter/setter) property named __proto__ 
    var object = { __proto__: anotherObject };
    // object.__proto__ = anotherObject

    // (5) Object.getPrototypeOf and Object.setPrototypeOf (get/set the [[Prototype]] after object creation)
    // (similar to 4.)
You can't actually see the link target (Object.prototype) in a new statement; instead the target is implied by the constructor (Object).

Remember:      Every object has a link, [[Prototype]], sometimes exposed as __proto__.
                        Every function has a prototype property, initially holding an empty object.
                        Objects created with new are linked to the prototype property of their constructor.
                        If a function is never used as a constructor, its prototype property will go unused.
                        If you don't need a constructor, use Object.create instead of new.

    // ES5:
    function Child() {}
    function Parent() {}
    Parent.prototype.inheritedMethod = function () { return 'this is inherited'; };

    // Object.create
    function inherit(child, parent) {
        child.prototype = Object.create(parent.prototype);
        child.prototype.constructor = child;
        return child;
    }

    // Alternatively without using Object.create
    function inherit(child, parent) {
        function tmp() {}
        tmp.prototype = parent.prototype;
        const proto = new tmp();
        proto.constructor = child;
        child.prototype = proto;
        return child;
    }

    Child = inherit(Child, Parent);
    const c = new Child;
    console.log(c.inheritedMethod()); // 'this is inherited'

The class syntax introduced in ES6 simplifies things, by providing extends as the "one true way" to configure the prototype chain in order to simulate classical inheritance in JavaScript.
    // ES6 - Easy way !
    class Parent { inheritedMethod() { return 'this is inherited' } }
    class Child extends Parent {}

    const c = new Child;
    console.log(c.inheritedMethod()); // 'this is inherited'

An anonymous function is just a function that has no name; nothing more. 
A closure is a function that captures the state of the surrounding environment.
An anonymous function does not necessarily need to create a closure, and a closure is not created only for anonymous functions.

Clousure -means- 2 thinks comes to your mind:
                     1. A function inside a function must exist.
                     2. Outer function must have at least one local variable. That local variable must be accessed by the inner function

Problem:
    for(var i = 0; i < 5; i++) {
        setTimeout(function() { console.log(i); }, 0); // 5 5 5 5 5
    }

    // Solution 1:
    for(var i = 0; i < 5; i++) {
        (function f() {
            var i2 = i; 
            setTimeout(function g() { console.log(i2); }, 0); // 0 1 2 3 4
        })();
    }
Note: 'f' is NOT a closure, but 'g' is a Closure
Why 'f' is NOT closure ?
Consider only the variable(s) which are from outer scope. In this case - i
'f' is NOT a closure bcos both 'i' and 'f' are referenced by the same global scope. 
NO state maintenance is needed.

why 'g' is a Closure ?
Consider only the variable(s) which are from outer scope. In this case - i2
'g' is a closure bcos both 'i2' and 'g' are in different scope. 'i2' is in 'f' scope and 'g' is in setTimeout scope.
So state should be maintained.

    // Solution 2:
    for(var i = 0; i < 5; i++) {

            return function g() { console.log(i2); } // 0 1 2 3 4
        })(i), 0);
    }
Note: 'f' is NOT a closure, but 'g' is Closure
Why 'f' is NOT closure ?
Consider only the variable(s) which are from outer scope. In this case - NOTHING.
so 'f' is NOT a closure.

why 'g' is a Closure ?
Consider only the variable(s) which are from outer scope. In this case - i2
'g' is closure bcos they are referenced in different scope. 'i2' is in 'f' scope and also in 'g' scope.
So state should be maintained.

    inc.update(); // 2

    var inc2 = new outerFunc();
    inc2.update(); // 1
    inc2.update(); // 2

A closure is a function that, when called, uses the scope in which it was declared, not the scope in which it was called. (Just opposite to 'this' context)
Capture & Return:
    function foo() {
        var x = 42;
        var inner  = function() { console.log(x); };
        x = x + 1;
        return inner; // Capture the final value at time of return from inner function
    }
    var f = foo();
    f(); // 43

Capture & Local variable:
    var z = 1;
    function x() {
        var z = 2;
        y(function(){ console.log(z); }); // z's value captured here
        z = 100; // NOT change the captured value
    }
    function y(f) {
        var z = 3;
        f();
    }
    x(); // 2 

Capture & Global variable:
    var z = 1;
    function x() {
        z = 2; // NOT local variable
        y(function(){ console.log(z); }); // z's value captured here
        z = 100;
    }
    function y(f) {
        f();   // 2 NOT 100
        z = 3; // overrides the captured global value here
        f();   // 3 NOT 2
    }
    x();

A closure is the combination of a function and the lexical environment within which that function was declared. This environment consists of any local variables that 
were in-scope at the time the closure was created.
Both produce same output:
    function init() {
        var name = 'Mozilla';    // name is a local variable created by init
        function displayName() { // displayName() is the inner function, a closure
            alert(name);         // use variable declared in the parent function
        }
        displayName();
    }
    init();

    function makeFunc() {
        var name = 'Mozilla';
        function displayName() {
            alert(name);
        }
        return displayName;
    }   
    var myFunc = makeFunc();
    myFunc();

A closure is a pairing of:
    1. A function, and
    2. A reference to that function's outer scope (lexical environment)
A lexical environment is part of every execution context (stack frame) of a function, and is a map between identifiers (ie. local variable names) and values.

Every function in JavaScript maintains a reference to its outer scope (lexical environment). This reference is used to configure the execution context(stack frame) created when a function is invoked. This reference enables code inside the function to "see" variables declared outside the function, regardless of when and where the function is called.

If a function was called by a function, which in turn was called by another function, then a chain of references to outer lexical environments is created. This chain is called the scope chain.

Every function in JavaScript maintains a link to its outer lexical environment. A lexical environment is a map of all the names (eg. variables, parameters) within a scope, with their values. So, whenever you see the function keyword, code inside that function has access to variables declared outside the function.

Closure various forms/ Keywords / Hints:
        1. Inner function: Function inside an another function (must)
        2. Function 'returns' another function (return keyword is optional)
        3. anonymous function inside an another function
        4. As a wrapper function
        5. whenever you need private state associated with a function. 
        6. new Function(…) (the Function constructor) inside a function NOT creates a closure. Bcos the new function cannot reference the local variables.

Performance considerations: 
It is unwise to unnecessarily create functions within other functions if closures are not needed for a particular task, as it will negatively affect script performance both in terms of processing speed and memory consumption.


Practical Use of Closures:
1. Enforce public/private methods
    a = (function () {
        var privatefunction = function () { alert('hello'); }
        return {
                        publicfunction : function () {
                                                        privatefunction();
                                                     }
               }
        })();
'a' is an object, with a method publicfunction which calls privatefunction, which only exists inside the closure. You can NOT call privatefunction directly 
(i.e. a.privatefunction() ), just as a.publicfunction().

2.Count the number of times user clicked a button on a webpage.
        1) You could use a global variable, and a function to increase the counter:
    <button onclick="updateClickCount()">click me</button>

Solution 1: Using global variable:
    var counter = 0;
    function updateClickCount() {
        ++counter;
        // do something with counter
    }
But, the pitfall is that any script on the page can change the counter, without calling updateClickCount().

Solution 2: Closure to the rescue! (self-invoking function):
    var updateClickCount = (function() {
                                            var counter = 0;
                                            return function() {
                                                                    ++counter;
                                                                    // do something with counter
                                                               }
                                        })();
The self-invoking function only runs once. It sets the counter to zero (0), and returns a function reference.
By JavaScript closure, It makes possible for a function to have "private" variables. 
The counter is protected by the scope of the anonymous function, and can only be changed using the add function!

Live Example:

        var updateClickCount = (function() {
            var counter = 0;
            return function() {
                ++counter;
                document.getElementById("spnCount").innerHTML=counter;
            }
        })();
    </script>

    <html>
    <button onclick="updateClickCount()">click me</button>
    <div> you've clicked 
        <span id="spnCount"> 0 </span> times!
    </div>
    </html>

3.  Closure & Callback: You can use a closure anywhere that you might normally use an object with only a single method.
Situations where you might want to do this are particularly common on the web. Much of the code written in front-end JavaScript is event-based. You define some behavior, and then attach it to an event that is triggered by the user (such as a click or a keypress). The code is attached as a callback (a single function that is executed in response to the event).
    function makeSizer(size) {
        return function() {
            document.body.style.fontSize = size + 'px';
        };
    }

    var size12 = makeSizer(12);
    var size14 = makeSizer(14);
    var size16 = makeSizer(16);

    document.getElementById('size-12').onclick = size12;
    document.getElementById('size-14').onclick = size14;
    document.getElementById('size-16').onclick = size16;

Throttling and Debouncing: Both help you postpone and reduce the rate of some execution. 

Assuming you are calling decorated functions returned by throttle/debounce repeatedly...
Throttle: the original function be called at most once per specified period.
Throttling will delay executing a function. It will reduce the notifications of an event that fires multiple times.

Debounce: the original function be called after the caller stops calling the decorated function after a specified period.
Debouncing will bunch a series of sequential calls to a function into a single call to that function. It ensures that one notification is made for an event that fires multiple times.

3. Throttling:
    const throttle = (func, limit) => {
        let isThrottling;
        return function() {
                    const context = this;
                    const args = arguments;
                    if (!isThrottling) {
                        func.apply(context, args);
                        isThrottling = true;
                        setTimeout(() => isThrottling = false, limit);
                    }
                }
    }

4. Debouncing:
    const debounce = (func, delay) => {
        let debouncing;
        return function() {
                    const context = this;
                    const args = arguments;
                    clearTimeout(debouncing);
                    debouncing = setTimeout(() => func.apply(context, args), delay);
                }
    }


If you have a function that gets called a lot - for example when a resize or mouse move event occurs, it can be called a lot of times. If you don't want this behavior, you can Throttle it so that the function is called at regular intervals. Debouncing will mean it is called at the end (or start) of a bunch of events.


Use cases for debounce:
    Typing: You want to do something after the user stopped typing. So waiting 1sec after the last keystroke makes sense. Each keystroke restarts the wait.
    Animation: You want to shrink back an element after the user stopped hovering over it. Not using debounce might cause an erratic animation as a result of the cursor 
                        unintentionally moving between the "hot" and "cold" zones.

Use cases for throttle:
    Scrolling: You want to react to scrolling but limit the amount of calculations made, so doing something every 100ms is enough to prevent potential lag.
    Mouse move: Same as scrolling but for mouse move.
    API calls: You want to fire an API call on certain UI events but want to limit the number of API calls you make not to overload your server.

Debouncing: The debounce pattern lets us control events that are triggered successively, and if the interval within two events is less than a certain amount of time, it completely ignores the first one. This process is repeated until it gets a pause higher than the desired minimum interval. Once it occurs, just the last occurrence of the event before the pause will be executed, ignoring all the previous ones.
In plain words: Debouncing makes that a function can’t be invoked again until a specific set of time has given without it being called. 
Example: Execute a function only if 1000 milliseconds have passed without it being called.
