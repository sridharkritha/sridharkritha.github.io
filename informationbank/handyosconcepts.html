Inter-Process Communication

Inter-Process Communication (IPC) is a set of techniques for the exchange of data between two or more processes. Processes may be running on one or more computers connected by a network.

 

The IPC mechanisms can be classified into the following categories as given below:

 

   1. Pipes

   2. Fifo

   3. Shared memory

   4. Message queues

   5. Sockets

 

 

1. Pipes: They provide unidirectional (half-duplex) flow of communication between processes within the same system. The limitation of pipes is that only processes 'related' to the process that created the pipe (i.e. siblings of that process) may communicate using them. If we want, two un-related processes to communicate via pipes, we need to use named pipes.

 

2. FIFO: FIFOs (first in, first out) are similar to the working of pipes. FIFOs also provide half-duplex flow of data just like pipes. The difference between fifo and pipes is that fifo are named pipes ie., Fifo are identified by an access point which is a file kept on the file system, whereas pipes are identified by an access point which is simply an allotted inode. Another major difference between fifos and pipes is that fifo last throughout the life-cycle of the system, while pipes last only during the life-cycle of the process in which they were created.

 

3.Shared Memory: Shared Memory which enables different processes to communicate with each other as if these processes shared the virtual address space; hence, any process sharing the memory region can read or write to it.

 

4.Message queues: It  provide an asynchronous communications protocol, meaning that the sender and receiver of the message do not need to connect to the message queue at the same time. Messages placed onto the queue are stored until the recipient retrieves them. Most message queues have set limits on the size of data that can be transmitted in a single message. Those that do not have such limits are known as mailboxes.

 

5. Sockets: Sockets provide point-to-point, two-way communication between two processes. A socket is an endpoint of communication to which a name can be bound.  

Socket types define the communication properties visible to the application. Processes communicate only between sockets of the same type. There are five types of socket.  

A stream socket

-- provides two-way, sequenced, reliable, and unduplicated flow of data with no record boundaries. A stream operates much like a telephone conversation. The socket type is SOCK_STREAM, which, in the Internet domain, uses Transmission Control Protocol (TCP).

A datagram socket

-- supports a two-way flow of messages. Datagram socket may receive messages in a different order from the sequence in which the messages were sent. Record boundaries in the data are preserved. Datagram sockets operate much like passing letters back and forth in the mail. The socket type is SOCK_DGRAM, which, in the Internet domain, uses User Datagram Protocol (UDP).

A sequential packet socket

-- provides a two-way, sequenced, reliable, connection, for datagram’s of a fixed maximum length. The socket type is SOCK_SEQPACKET. No protocol for this type has been implemented for any protocol family.

A raw socket

            -- Raw sockets allow you to bypass some of the way of encapsulation/decapsulation that the TCP/IP stack on the kernel does, you just pass the packet to the application that needs it. No TCP/IP processing was done -- so it's not a processed packet, it's a raw packet. The application that's using the packet is now responsible for stripping off the headers, analyzing the packet, and do all the stuff that the TCP/IP stack in the kernel normally does for you. ie., raw socket is a socket that takes packets, bypasses the normal TCP/IP processing, and sends them to the application that wants them.

Semaphores

Different types of semaphores?

1. Mutex Semaphores 2. Binary Semaphore and 3. Counting Semaphores (normally called as simply Semaphores).

Difference between Mutex and Binary Semaphore?

 

Mutex is actually an extension of binary semaphore with some extra features. If we are not using the extra features, this can be considered as a binary semaphore.

 

Differences are: 

1. If Mutex is acquired by some process, only that process can unlock it, where as a binary semaphore can be unlocked by any other process

 

2. In Mutex, recursive calling is possible (ie., Mutex can be taken more than once by the task that holds it before finally being released.), where as this feature is not available in binary semaphore.

 

3. Priority inversion is possible with a Mutex semaphore. The same is not applicable in case of binary semaphore

4. Mutex cannot be given from an ISR module.

5. Mutex provides an option for making the task that took it as DELETE_SAFE. This means, that the task cannot be deleted when it holds Mutex.

The Toilet Example

Mutex:

 

Is a key to a toilet. One person can have the key - occupy the toilet - at the time. When finished, the person gives (frees) the key to the next person in the queue.

 

Officially: "Mutexes are typically used to serialize access to a section of re-entrant code that cannot be executed concurrently by more than one thread. A mutex object only allows one thread into a controlled section, forcing other threads which attempt to gain access to that section to wait until the first thread has exited from that section." 

(A mutex is really a semaphore with value 1.)

 

Semaphore:

 

Is the number of free identical toilet keys. Example, say we have four toilets with identical locks and keys. The semaphore count - the count of keys - is set to 4 at beginning (all four toilets are free), then the count value is decremented as people are coming in. If all toilets are full, i.e. there are no free keys left, the semaphore count is 0. Now, when eq. one person leaves the toilet, semaphore is increased to 1 (one free key), and given to the next person in the queue.

 

Officially: "A semaphore restricts the number of simultaneous users of a shared resource up to a maximum number. Threads can request access to the resource (decrementing the semaphore), and can signal that they have finished using the resource (incrementing the semaphore)."

Producer-consumer:
Whenever an event occurs, a producer thread creates an event object and adds it to the event buffer. Concurrently, consumer threads take events out of the buffer and process them. In this case, the consumers are called “event handlers.”


1. If a producer arrives when the buffer is full, it blocks until a consumer removes an item.
2. If a consumer thread arrives while the buffer is empty, it blocks until a producer adds a new item
3. Imagine that the buffer is empty. A consumer arrives, gets the mutex, and then blocks onitems. When the producer arrives, it blocks on mutex and the system becomes deadlock.

A.  Producer-consumer initialization (Finite-buffer)
1 items = Semaphore(0)
2 spaces = Semaphore(buffer.size())

3 mutex = Semaphore(1)


B. Producer:
1 event = waitForEvent()
2
3 spaces.wait()
4 mutex.wait()
5   buffer.add(event)
6 mutex.signal()
7 items.signal()


C. Consumer:
1 items.wait()
2 mutex.wait()
3   event = buffer.get()
4 mutex.signal()
5 spaces.signal()
6
7 event.process()

class BlockingQueue {
    Queue<String> buffer = new LinkedList<String>();

    public void Producer(String data) {

     synchronized(lock){
        buffer.add(data);
        notify();                   // Since someone may be waiting in take!
         }
     }

   public String Consumer() throws InterruptedException {
      
    synchronized(lock){
        while (buffer.isEmpty()) {  // don't use "if" due to spurious wakeups.
            wait();
           }
        return buffer.remove();
    }
}

// ReadWriteThread.cpp : Defines the entry point for the console application.
//

#include "stdafx.h"
// int _tmain(int argc, _TCHAR* argv[]) { return 0; }

#include <stdio.h>
#include <conio.h>
#include <windows.h>

HANDLE mutex, wrt;
int g_ReadCount = 0;
int g_GlobalData = 0;
const int max = 3;
HANDLE Threads[max];
CRITICAL_SECTION rSect, wSect;
bool bTerminate = true;

DWORD Readers(LPVOID lpdwThreadParam )
{
	while(bTerminate)
	{
		WaitForSingleObject(mutex, INFINITE);
		g_ReadCount++;
		if(g_ReadCount == 1)
		{
			WaitForSingleObject(wrt, INFINITE);
		}
		ReleaseMutex(mutex);

		EnterCriticalSection(&wSect);
		printf("READER - ThreadId : %d --> Read data : %d ReaderCount %d\n", GetCurrentThreadId(), g_GlobalData, g_ReadCount);
		LeaveCriticalSection(&wSect);

		WaitForSingleObject(mutex, INFINITE);
		g_ReadCount--;
		if(g_ReadCount == 0)
		{
			ReleaseMutex(wrt);
			printf("READER - ThreadId : %d Realesed Mutex wrt\n", GetCurrentThreadId());
		}
		printf("READER - ThreadId : %d ReaderCount %d\n", GetCurrentThreadId(), g_ReadCount);
		ReleaseMutex(mutex);
		printf("READER - ThreadId : %d Realesed Mutex mutex\n", g_ReadCount);
		Sleep(0);
	}
	printf("READER - Exit\n");
	return 0;
}


DWORD Writers(LPVOID lpdwThreadParam )
{
	while(bTerminate)
	{
		printf("WRITER - ThreadId : %d Waiting for WRT\n", GetCurrentThreadId());
		WaitForSingleObject(wrt, INFINITE);
		printf("WRITER - ThreadId : %d ***Got WRT\n", GetCurrentThreadId());

		EnterCriticalSection(&wSect);
		g_GlobalData = g_GlobalData + 1;
		printf("WRITER - ThreadId : %d Updated Data: %d \n", GetCurrentThreadId(), g_GlobalData);
		if(5 == g_GlobalData)
			bTerminate = false;
		LeaveCriticalSection(&wSect);
		ReleaseMutex(wrt);
	}

	printf("WRITER - Exit\n");
	return 0;
}

void main()
{
	mutex = CreateMutex(NULL, false, "Writer");
	wrt = CreateMutex(NULL, false, "wrt");

	InitializeCriticalSection(&rSect);
	InitializeCriticalSection(&wSect);
	DWORD dwThreadId = 0;

	// Writer Thread #1
	Threads[0] = CreateThread(NULL, //Choose default security
		0, //Default stack size
		(LPTHREAD_START_ROUTINE)&Writers,
		//Routine to execute
		(LPVOID) 0, //Thread parameter
		0, //Immediately run the thread
		&dwThreadId //Thread Id
		);


	// Reader Threads #2
	for(int i=1; i <= 2; i++)
	{
		Threads[i] = CreateThread(NULL, //Choose default security
			0, //Default stack size
			(LPTHREAD_START_ROUTINE)&Readers,
			//Routine to execute
			(LPVOID) 0, //Thread parameter
			0, //Immediately run the thread
			&dwThreadId //Thread Id
			);
	}

	// Blocks/waits till all child threads are finished
	// If 3rd TRUE, the function returns when the state of all objects in the handle array is signaled.
	// If FALSE, the function returns when the state of any one of the objects is set to signaled
	// The INFINITE - the function will return only when the specified objects are signaled.
	WaitForMultipleObjects(_countof(Threads), Threads,TRUE, INFINITE);

	// close all handles to threads
	for(int i=0;i< max /*THREADCOUNT*/;i++)
	{
		if(CloseHandle(Threads[i]) != 0)
			wprintf(L"hTr's handle was closed successfully!\n");
		else
			wprintf(L"Failed to close hTr's handle, error %u\n", GetLastError());
	}

	getchar();
}

Threads Vs Processes is:

  1. Threads share the address space of the process that created it(except stack and local variables); processes have their own address space.
  2. Threads have direct access to the data segment of its process; processes have their own copy of the data segment of the parent process.
  3. Threads can directly communicate with other threads of its process; processes must use interprocess communication to communicate with sibling processes.
  4. Threads have almost no overhead; processes have considerable overhead.
  5. New threads are easily created; new processes require duplication of the parent process.
  6. Threads can exercise considerable control over threads of the same process; processes can only exercise control over child processes.
  7. Changes to the main thread (cancellation, priority change, etc.) may affect the behavior of the other threads of the process; changes to the parent process does not affect child processes.
  8. Allocation space (kernel space for processes vs. user space for threads)
