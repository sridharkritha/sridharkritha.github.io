emplace Vs insert:
1. Emplace takes the arguments necessary to construct an object in place, whereas 
    insert takes (a reference to) an object.

2. insert copies objects into the vector. C++11 insert does not have to copy, it can also move.
    emplace construct them inside of the vector.
    struct Foo
    {
        Foo(int n, double x);
    };

    std::vector<Foo> v;
    v.emplace(someIterator, 42, 3.1416);
    v.insert(someIterator, Foo(42, 3.1416));    

Summary:

Sequence containers: Sequence containers implement data structures that can be accessed sequentially.
    array        - static contiguous array
    vector       - dynamic contiguous array
    deque        - double-ended queue
    forward_list - singly-linked list
    list         - doubly-linked list

Associative containers: Associative containers implement sorted data structures that can be quickly searched in O(log n).
    set      - collection of unique keys, sorted by keys
    map      - collection of key-value pairs, sorted by keys, keys are unique
    multiset - collection of keys, sorted by keys
    multimap - collection of key-value pairs, sorted by keys

Unordered associative containers: Unordered associative containers implement unsorted (hashed) data structures that can be quickly searched (O(1) amortized, O(n) worst-case complexity).
    unordered_set      - collection of unique keys, hashed by keys
    unordered_map      - collection of key-value pairs, hashed by keys, keys are unique
    unordered_multiset - collection of keys, hashed by keys
    unordered_multimap - collection of key-value pairs, hashed by keys

Container adaptors: Container adaptors provide a different interface for sequential containers.
    stack          - adapts a container to provide stack (LIFO data structure)
    queue          - adapts a container to provide queue (FIFO data structure)
    priority_queue - adapts a container to provide priority queue

Methods:
A. Sequence containers:
    vector             [push_back, pop_back, front, back]
    deque              [push_back, pop_back, front, back, push_front, pop_front]
    list               [push_back, pop_back, front, back, push_front, pop_front]
forward_list       [  front,  push_front, pop_front]

B. Associative containers:
    set                [insert, count, find, lower_bound]
    multiset           [insert, count, find, lower_bound]
    map                [insert, count, find, lower_bound, operator[] ]
    multimap           [insert, count, find, lower_bound]

C. Unordered Associative container:
    unordered_set      [insert, count, find]
    unordered_multiset [insert, count, find]
    unordered_map      [insert, count, find, operator[] ]
    unordered_multimap [insert, count, find]

D. Container adapters:
    stack              [top,         pop, push]
    queue              [front, back, pop, push]
    priority_queue     [top,         pop, push]

How to clear stack container adaptor?
Container adaptors(stack, queue, priority_queue) does NOT support clear() so best way to clear the container adaptors (or any other container) is assign to empty or swap to empty container.
    stack<int> s = {1,2,3,4,5};
    // Method 1: Recommended
    s = {}; // clears all the content
    // Method 2: Idiom for clearing standard containers is swapping with an empty version of the container:
    std::queue<int> empty;
    std::swap( s, empty );

Notes:
    1. Inserting / Erasing on "vector" will INVALIDATES the previous value of pointers / reference / iterator.
  2. Inserting on "deque" on both ends will NOT INVALIDATES but INVALIDATES when insertion on the middle.
    3. Inserting / Erasing on "list, Associative container, Unordered container" will NOT invalidates the 
       previous value of pointers / reference / iterator.

    vec.push_back(6);  // inset/delete will do reallocation - all pointer/ref/itr becomes invalid
    deq.push_back(6);  // OK - inserting at either END won't invalidate pointers but invalidates insertion at MIDDLE
    deq.push_front(6); // OK - do -

multimap: Duplicate key allowed. The same key can have different values. Keys are sorted (NOT values).

Finding all values for a key in multimap?
    equal_range: Can be used to print all the different values of the given key.
    multimap<char, int> myMaplist = { {'c', 5}, {'a',6}, {'b',3}, {'a',4}, {'a',7}, };
    for (auto itr : myMaplist) cout << itr.first << " " << itr.second << endl; // Hint: -> NOT needed for here
    // a 6    a 4     a 7     b 3     c 5

    std::pair<std::multimap<char, int>::iterator, std::multimap<char, int>::iterator> ret;
    ret = myMaplist.equal_range('a');

   // equal_range: Print all the different values for a given duplicate key 'a'
   for(std::multimap<char, int>::iterator it = ret.first; it != ret.second; ++it) //'first' compared with 'second'
        std::cout << it->first <<  " " << it->second; // Hint: -> Needed for here
   // a 6     a 4     a 7   

    // Delete all values of a key 'a'
    for (std::multimap<char, int>::iterator it = ret.first; it != ret.second;)
    {
        std::multimap<char, int>::iterator tempItr = it++; // MUST bcos erase invalidates the iterator
        myMaplist.erase(tempItr); // deleted => a 6     a 4     a 7  
    }

    it = vec.begin(); // it -> 12
    it = vec.end();   // it -> X after 18

Search by POSITION/ITERATOR:
    vector<int> vec = { 2,9,9,9,62,87,90 };      // MUST be sorted for lower_bound usage

    auto itr = lower_bound(vec.begin(), vec.end(), 9); // returns iterator pointing to vec[1] which is 9.  
    cout << itr - vec.begin(); // index = 1
    // Finds the first position(lower index) where 9 could be inserted and still keep the sorting.
    vec.insert(itr, 7);  // 2,7,9,9,9,62,87,90

    itr = upper_bound(vec.begin(), vec.end(), 9);      // returns iterator pointing to vec[5] which is 62.  
    cout << itr - vec.begin(); // index = 5
    // Finds the last position(upper index) where 9 could be inserted and still keep the sorting.
    vec.insert(itr, 44);  // 2,7,9,9,9,44,62,87,90

Input and output iterators are the most limited types of iterators: they can perform sequential single-pass input or output operations.

Forward iterators have all the functionality of input iterators and output iterators, although they are limited to one direction in which to iterate through a range (forward). All standard containers support at least forward iterator types.

Bidirectional iterators are like forward iterators but can also be iterated backwards.

Random-access iterators implement all the functionality of bidirectional iterators, and also have the ability to access ranges non-sequentially: distant elements can be accessed directly by applying an offset value to an iterator without iterating through all the elements in between. These iterators have a similar functionality to standard pointers (pointers are iterators of this category).

      // First erase OK; second one is undefined behavior
   } 
}

// Solution:
for (itr=s.begin(); itr!=s.end(); ) // Loop alteration: MOVE the itr increment to the bottom
{
   if (*itr == 1) { // Remove all 1
      cout << "Erase one item of " << *itr << endl; // OK - moved above erase()
      // postfix increment returns old itr for to delete the item and advance to next element.
      s.erase(itr++); // itr points to item to be deleted and becomes invalid, itr++ points to next element.
      // {4, 12, 18, 16}
   } else {
      itr++;
   }
}

// 2. Sequence Container:
vector<int> v = {1, 4, 1, 1, 1, 12, 18, 16};
vector<int>::iterator itr2;
// Problem:
for (itr2=v.begin(); itr2!=v.end(); ) {
   if (*itr2 == 1) { // Remove all 1
      cout << "Erase one item of " << *itr2 << endl;
      // In ALL sequence containers - if an element is erased then all the iterators AFTER that erased element 
// will be INVALID.
      v.erase(itr2++);// Eventhough itr++ moved to next element, it is invalid/undefined behaviour for the sequence containers.
      // {4, 1, 12, 18, 16} - WRONG - Undefined behaviour
   } else {
      itr2++;
   }
}

// Solution:
for (itr2=v.begin(); itr2!=v.end(); ) {
   if (*itr2 == 1) { // Remove all 1
      cout << "Erase one item of " << *itr2 << endl;
      // v.erase(itr2++);   // Wouldn't work correctly also DONOT CRASH - very bad error
      itr2 = v.erase(itr2); // OK. Sequence container and unordered container's erase() returns  
                            // iterator pointing to next item after the erased item.
   } else {
      itr2++;
   }
}

Summary:
        1. Sequence container and Unordered container's erase() returns the next iterator after the erased item.
                    itr = v.erase(itr);
        2. Associative container's erase() returns nothing.
                    v.erase(itr++)

    // vector
    // Ref: Learn STL: Vector vs. Deque - part I

    int main() {
        vector<int> vec = {2,3,4,5};
        cout<< vec.size << vec.capacity(); // vec.size() == 4, vec.capacity() == 4, 
        vec.push_back(6); // Capcity is full so reallocation will happen
        cout<< vec.size << vec.capacity(); // vec.size() == 5, vec.capacity() == 8 (NOT 5)  
        // Reallocation happens(mostly double the capacity of old one) whenever the capacity is full.
    }


    class Dog;
    
    int main() {
        // Example 1:
        vector<Dog> vec(6); // vec.capacity() == 6, vec.size() == 6, 
                            // 6 Dogs created with default constructor
        // Example 2:
        vector<Dog> vec;  // vec.capacity() == 0, vec.size() == 0
        vec.resize(6);    // vec.capacity() == 6, vec.size() == 6,  
                         // 6 Dogs created with default constructor 
        // Example 3:
        vector<Dog> vec; // vec.capacity() == 0, vec.size() == 0
        vec.reserve(6);   // vec.capacity() == 6, vec.size() == 0 (NOT 6), 
                          // no default constructor invoked
    }

    The strategy for avoiding reallocation:
        1. If the maximum number of item is known, use reserve(MAX);
        2. If unknown, reserve as much as you can, once all data inserted, trim off the rest.

    // For trim off:
    v.shrink_to_fit();      // C++ 11
    vector<int>(c).swap(c); // C++ 03

     Advantages of deque :  - No reallocation 
                            - deque has NO reserve() and capacity()
                            - But slightly slower than vector

Prefer deque if - 
1. Element data type: When the elements are not built-in type, the deque is not much less efficient than a vector. 
            Mostly all container behave a bit less performance than usual.
2. Memory Availability:
Could the allocation of large contiguous memory be a problem? 
- Limited memory size
- Large trunk of data
3. Frequency of Unpredictable Growth:
    vector<int> vec;
    for (int x=0; x<1025; x++) 
        vec.push_back(x);   // 11 reallocations performed (growth ratio = 2)
                            // workaround: reserve()
4. Invalidation of pointers/references/iterators because of growth
    vector<int> vec = {2,3,4,5}; 
    int* p = &vec[3]
    vec.push_back(6);   // Reallocation happens - all pointer/ref/itr becomes invalid
    cout << *p << endl; // Undefined behavior

    deque<int> deq = {2,3,4,5};
    p = &deq[3];
    deq.push_back(6);
    cout << *p << endl;  // OK
    // push_front() is OK too
    // deque: inserting at either END won't invalidate pointers
    // Note: removing or inserting in the MIDDLE still will invalidate pointers/references/iterators
5. Vector's unique feature: Portable to C
    // Passing data from a vector to C
    vector<int> vec = {2,3,4,5};    
    c_fun(&vec[0], vec.size()); // void c_fun(const int* arr, int size) { }

    // Passing data from a list to C
    list<int> mylist;
    // All other containers need to copy on vector before passing to C function
    vector<int> vec(mylist.begin(), mylist.end()); 
    c_fun(&vec[0], vec.size());

    // NOTE: &vector[0] can be used as a raw array.
    // Exception: vector<bool> 
    vector<bool> vec = {true, true, false, true}; 
    void cpp_fun(const bool* arr, int size); // bool type is NOT supported in C
    cpp_fun(&vec[0], vec.size()); // Compiler Error: &vec[0] is not a bool pointer bcos it optimized to bit values
                                  // workaround: use vector<int>, or bitset

    Summary:
        1. Frequent push_front()    - deque
        2. High performance         - vector
        3. Non-build in data type   - deque
        4. No Contiguous memory     - deque
        5. Unpredictable growth     - deque
        6. Pointer integrity        - deque
        7. Talk to C                - vector

What is object slicing in STL containers ? Some part of the object being sliced while doing an object copy.
    class Dog {
    public:
        void bark() { cout << "I don't have a name"; } // NOT a virtual function
    };

    class YellowDog : public Dog {
        string mName;
    public:
        YellowDog(string name) : mName(name) { }
        void bark() { cout << "My name is " << mName; }
    };

    void bar(Dog d)  // Problem 2: Dog is copied NOT YellowDog. ONLY sliced version is copied.
    {
        d.bark();   // I don't have a name
    }
    void foo(YellowDog d)
    {
        d.bark();         // My name is Gunner
    }

    int main() {
        YellowDog y("Gunner");

        deque<Dog> d;
        // STL containers always copy objects bcos of passing by value.
        // Also 'Pass by value' cause a 'static cast' that slice the object. 
        d.push_front(y); // y is NOT pushed instead the copy constructed Dog is pushed. Sliced version of 'y' 
                         // is pushed.
        d[0].bark();     // I don't have a name [ PROBLEM 1 ]

        bar(y); // [ PROBLEM 2] 
        foo(y); // OK
    }



Solution:
// Problem 1:
    // Step 1: Use Reference/Pointer
    int main() {
        YellowDog y("Gunner");
        // Change to Pointer
        deque<Dog*> d;
        d.push_front(&y); // NOT plain 'y' (Slicing avoided)
        d[0]->bark();     // My name is Gunner
    }

    class Dog {

    };

// be casted dynamically 
        YellowDog* py = dynamic_cast<YellowDog*>(d); // Avoid static cast
        cout << py->mName << endl;  // Gunner
    }
