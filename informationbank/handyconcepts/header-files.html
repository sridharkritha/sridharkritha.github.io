Header Files

Ref: http://www.cplusplus.com/forum/articles/10627/#msg49679

Declaration Vs Definition:

A declaration introduces an identifier and describes its type. A declaration is what the compiler needs to accept references to that identifier. These are declarations:

    static int bar;           // declares bar
    extern int bar;           // declares bar
    extern int fun(int, int); // declares fun
    int fun(int, int);        // extern can be omitted for function declarations

    class C;                  // no extern allowed for class declarations
    struct S;                 // declares S
    typedef long LONG_32;     // declares LONG_32
    using namespace std;      // declares std
    
    extern "C"
    {
        void foo();           // declares foo
    }

A definition actually instantiates/implements this identifier. It's what the linker needs in order to link references to those entities. These are definitions corresponding to the above declarations:

    int bar;                   // defines bar
    extern const int bar = 0;  // defines bar
    int fun(int lhs, int rhs) {return lhs*rhs;}
    class C {};
    struct S { int a; int b; }; // defines S, S::a, and S::b
An identifier can be declared as often as you want. Thus, the following is legal in C and C++:
    double f(int, double);
    double f(int, double);
    extern double f(int, double); // the same as the two above
    extern double f(int, double);
However, it must be defined exactly once (One Definition Rule)

More Examples:
1.
    struct X {
        int a;         // defines a
        static int b;  // declares b
    };
2. when using templates
    T t;

That can be a definition or a declaration, depending on what type T is:

    typedef void T();
    T t; // declaration of function "t"


    struct X { 
        T t; // declaration of function "t".
    };


    typedef int T;
    T t; // definition of object "t".
3.
    template <typename T>
    struct X { 
        static int member; // declaration
    };

    template<typename T>
    int X<T>::member; // definition


    template<>
    int X<bool>::member; // declaration!

The last declaration was not a definition. It's the declaration of an explicit specialization of the static member of X<bool>. It tells the compiler: "If it comes to instantiating X<bool>::member, then don't instantiate the definition of the member from the primary template, but use the definition found elsewhere". To make it a definition, you have to supply an initializer

    template<>
    int X<bool>::member = 1; // definition, belongs into a .cpp file.
HEADER FILE STRUCTURE :
Don't follow the advice of - not to put #include in header files just for avoiding the duplicate file inclusion. Actually, #including with guard in header files minimize repeated addition of include files in a family of headers.

Header Format:
    //=================================
    // include guard
    #ifndef __MYCLASS_H_INCLUDED__
    #define __MYCLASS_H_INCLUDED__

    //=================================
    // forward declared dependencies
    class Foo;
    class Bar;

    //=================================
    // included dependencies
    #include <vector>
    #include "parent.h"

    //=================================
    // the actual class
    class MyClass : public Parent  // Parent object, so #include "parent.h"
    {
     public:
        std::vector<int> avector;    // vector object, so #include <vector>
        Foo* foo;                    // Foo pointer, so forward declare Foo
        void Func(Bar& bar);         // Bar reference, so forward declare Bar

        friend class MyFriend;       // friend declaration is not a dependency
                                    //   don't do anything about MyFriend
    };

    #endif // __MYCLASS_H_INCLUDED__ 

Note: MyClass only uses a pointer to Foo and not a full Foo object, we can forward declare Foo, and don't need to #include "foo.h". 
 

Forward Declaration Vs. Include Header file:

There are two ways to resolve the dependencies of variable/function/parent class data types :

1)  stuff that can be resolved forward declared

2)  stuff that needs to be resolved #included

Note: 
1. Prefer forward declaration over #includes whenever possible.  This allows you to break the cyclic header dependencies.
2. Ensure that every header file #includes all other necessary headers or forward-declarations necessary for independent compilation.
3. Enforces the 'standalone' nature of header files.  In the implementation file xxx.cpp for the header xxx.h, ensure that xxx.h is the first header included.  If the header is not self-contained at any time, 
   then compilation will fail.   It is a beautifully simple and effective rule.
4. Never have "using namespace" in header files [makes global namespace whoever including that header file]. They should only be allowed in implementation files.

For example, class A (DERIVED CLASS) uses class B (BASE CLASS), then class B is one of class A's dependencies.  Whether it can be forward declared or needs to be 
included depends on how 'B' is used within 'A':

- do nothing if:         'A' makes no references at all to 'B'

- do nothing if:         The only reference to 'B' is in a friend declaration

- forward declare B if:  'A' contains a 'B' pointer or reference:  B* myPtB; B& myRefB;   [Variable needs only partial definition about data type at compile time bcos 
                         of pointer and reference usage ]

- forward declare B if:  one or more functions has a 'B' object/pointer/reference as a parementer, or as a return type:  B MyFunction(B myb); [Functions also only needs 
                         partial definition at compile time]

- #include "b.h" if:     'B' is a parent class of 'A'

- #include "b.h" if:     'A' contains a 'B' object:   B  myb; [Variable need to know the completed information about the data type. That is data type qualifies for passing 
                         through sizeof function for finding the size of the data type. ]

Circular Dependencies:
Problem 1:
    // a.h -- assume it's guarded
    #include "b.h"
    class A { 
        B* b; 
    };

    // b.h -- assume it's guarded
    #include "a.h"
    class B {
        A* a 
    };

This is a circular inclusion (also called an infinite inclusion).

Proof:

    // a.cpp
    #include "a.h" 
    int main () {
    .........
    }

The compiler will do the following expansion:

    #include "a.h"  // start compiling a.h
            #include "b.h"  // start compiling b.h
                    #include "a.h"  // compilation of a.h skipped because it's guarded
            // resume compiling b.h
            class B { A* a };   // <--- ERROR, A is undeclared
Solution:
  Always forward declare when you're only using a pointer or reference.  Here, "a.h" file should not be #including b.h, but instead should just be forward declaring B. 
  Likewise, b.h file should be forward declaring A. If you make those changes, the problem is solved.

Problem 2:
    // a.h (guarded)
    #include "b.h"
    class A {
        B b;   // B is an object, can't be forward declared
    };

    // b.h (guarded)
    class B {
        A a;   // A is an object, can't be forward declared
    };

Solution:
 This situation is conceptually impossible.  There is a fundamental design flaw.  If A has a B object, and B has an A object, then A contains a B, which contains 
  another A, which contains another B etc, etc.  You have an infinite recursion problem, and either class is simply impossible to instantiate. The solution is to
 have one or both classes contain a pointer or reference to the other, rather than a full object. Then you can forward declare, and then you can get around the 
 circular inclusion problem.
