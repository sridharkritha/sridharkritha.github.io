Methods to solve the linking error LNK2019: unresolved external symbol / undefined reference error:

    ////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    // A class with destructor
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    struct D {  ~D(); };
    
    // Destructor should always be defined explicitly otherwise linker error will occur
    // error LNK2019: unresolved external symbol "public: __thiscall D::~D(void)" (??1D@@QAE@XZ) referenced in 
// function _main
    D::~D() { } // should be defined

    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    // A class with a virtual destructor
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    struct VD { virtual ~VD(); };
    
    // error LNK2019: unresolved external symbol "public: virtual __thiscall VD::~VD(void)" (??1VD@@UAE@XZ) referenced 
// in function _main
    VD::~VD() { } // should be defined
    
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    // A class with Regular Virtual function (not destructor) 
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    struct RVF { virtual void foo(); };
    
    // Must be defined otherwise
    // error LNK2001: unresolved external symbol "public: virtual void __thiscall RVF::foo(void)" (?foo@RVF@@UAEXXZ)
    void RVF::foo() { } // should be defined
    
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    // A class with pure virtual destructor (Abstract Class)
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    struct PVD { virtual ~PVD() = 0; };
    
    // Must be defined otherwise
    // error LNK2019: unresolved external symbol "public: virtual __thiscall PVD::~PVD(void)" (??1PVD@@UAE@XZ) referenced
    // in function "public: virtual __thiscall Derived_PVD::~Derived_PVD(void)" (??1Derived_PVD@@UAE@XZ)
    PVD::~PVD(){ } // should be defined even it is pure virtual destructor
    
    struct Derived_PVD : PVD { };
    
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    // A class with Regular Pure Virtual Function (not destructor)
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    struct RPVF { virtual void bar() = 0; };
    
    struct Derived_RPVF : RPVF {
        void bar();
    };
    
    // Must be defined otherwise
    // error LNK2001: unresolved external symbol "public: virtual void __thiscall Derived_RPVF::bar(void)" 
// (?bar@Derived_RPVF@@UAEXXZ)
    void Derived_RPVF::bar() { }
    
    /////////////////////////////////////////////////////////////////////////////////////////////////////////////////////
    int main() {
        D d;
        VD vd;
        RVF rvf;             // Although there was no call to RVF::foo
        // PVD pvd;          // Abstract class cannot be instantiate
        Derived_PVD d_pvd;
        // RPVF rpvf;        // Abstract class cannot be instantiate
        Derived_RPVF d_rpvf; // Although there was no call to Derived_RPVF::bar
        return 0;
    }


Constructors: Special type member function whose function is to initialize the Object of its Class. A constructor does not have a return type, not even “Void” except the keyword "explicit'.

Class without default constructor:
    class base {
    public:
        base(int a) { std::cout << "base Single arg constructor" <<endl; }
    };

    int main() {
        base b; // Error: No default constructor - need to call the arg constructor explicitly [ base b(3); ]
    }    

Class with default and argument constructor:
    class base {
    public:        
        base() {  std::cout<< "base NO arg constructor" <<endl; } // Default constructor will not be called
        base(int a) { std::cout<< "base Single arg constructor" <<endl; }
    };

    int main() {
        base b(3); // Only calls the argument constructor not calls default constructor 
        return 0;
    }

Output: 
    base Single arg constructor

Constructors in Derived Classes: 
   The derived class need not have a constructor as long as the base class has no-argument constructor. However, if the base class has only constructors with arguments(one or more), then it is mandatory for the derived class to have a constructor and pass the arguments to the base class constructor.

Case I: Constuctor only in the derived class not in the base class:
    class B {
    // body of base class without constructors
    };

    class D: public B {
    public:
        D() { cout<< "Derived class no-argument constructor"; }
    };

    void main() {
        D obj_d;  // Derived class no-argument constructor
    }

In the above case, when the object of D is created the compiler will generate and call the default no argument constructor in the base class.

Case II: Multiple constructors in base class and a single constructor in derived class:
    class B {
    public:
        B() { cout<<"No argument Base Constructor"; }       
        B(int a) { cout<<"one argument Base Constructor"; }  // This will not be called automatic.
    };

    class D: public B {
    public:
        D(int a) { cout<<"One argument Derived Constructor"; }
    };

    void main() {
        D obj_d(5); 
    }

Output:
    No argument Base Constructor
    One argument Derived Constructor

Case III: Only argument constructor in base and derived class: - Need explicit invocation of constructors
    class B {
    public:
        // Called by explicit call [ : B(a) ] from derived class
        B(int a) { cout<<"One argument Base Constructor"; } // Compiler will NOT generate default no-argument constructor
    };

    class D: public B
    {
    public:
        D(int a) : B(a)  // if B(a) not exits then this program give compilation error - cannot find default constructor
        {
            cout<<"One argument Derived Constructor";
        }
    };

    void main() {
        D obj_d(5);
    }

Output:
    One argument Base Constructor
    One argument Derived Constructor

Case IV: Constructor in multiple inherited class with explicit invocation:
    class B1 {
    public:
        B1() { cout<<"B1: No argument Constructor"; }
    };

    class B2 {
        B2() { cout<<"B2: No argument Constructor"; }
    };

    class B3 {
        B3() { cout<<"B3: No argument Constructor"; }
    };

// order of inheritance - left to right but virtual have high priority than order
    class D: public B2, public B1, virtual B3 
    {
    public:
        D(): B1(), B3(),B2() // Explicit call to constructors
        {
            cout<<"D: No argument Constructor";
        }
    };

    void main() {
        D obj_d;  
    }

Output:
    B3: No argument Constructor    // because declared as virtual
    B2: No argument Constructor    // B2 declared before B1
    B1: No argument Constructor
    D:  No argument Constructor

Note: Order of invocation of constructors depends on the order of inheritance declaration but if it has virtual inheritance then it has more priority than others. Also, the order of constructor invocation does not follow the order of explicit call to the constructors.

Different ways of object declaration: B obj; B obj(); B obj(5); B obj = 5;
    class B {
    public:
        B()           { cout<<"Base::Constructor\n"; }
        B(int a)      { cout<<"Base::Single Arg Constructor\n"; }
        B(B const &b) { cout<<"Base::Copy Constructor\n"; }
        B& operator=(B const &b) { cout<<"Base::Overloaded Assignment\n"; }
        void t()      { cout<<"t function\n"; }
    };

    int main(){
        B t;     // Base::Constructor
        B t();   // Nothing: Function Declaration - function t() return data of type B.
        B t(5);  // Base::Single Arg Constructor - Preferred
        B t = 5; // Base::Single Arg Constructor - same as before
        B u = t; // Base::Copy Constructor
        B v;     // Base::Constructor
        v = t;   // Base::Overloaded Assignment
    }

Delete / free :
what is the use of the delete operator in c++?  Delete = free + destructor
So, If u want to free dynamically allocated memory, u can simply use free. But if u want to call destructor as well u  have to use delete. 

 Case 1:     
    char * lstrString = (char *) malloc (6 * sizeof(char));
    strcpy(lstrString, "12345");
    In this case, I have allocated 6 Bytes on heap for lstrString.
    Now, to free this memory, I can use     
    free(lstrString);
    lstrString = NULL;
    
 Case 2:     
    class A {
    public:
        char *lstrString;
        A(char * pstrString) {
            lstrString = (char *) malloc (strlen((pstrString)+1)  *  sizeof(char));
            strcpy(lstrString,  pstrString);
        }
        ~A() {
            free(lstrString);
            lstrString = NULL;
        }
    }

    int main() {
        A obj("Beba Spain");
        delete (obj);
    }

Faqs:

1. Named Constructor Idiom: The problem is that constructors always have the same name as the class. Therefore the only way to differentiate between the various constructors of a class is by the parameter list. But if there are lots of constructors, the differences between them become somewhat subtle and error-prone. With the Named Constructor Idiom, you can declare all the class's constructors with public static methods that return an object. These static methods are so-called "Named Constructors."
    class Point {
    public:
        Point(float x, float y);     // Rectangular coordinates
        Point(float r, float a);     // Polar coordinates (radius and angle)
        // ERROR: Overload is Ambiguous: Point::Point(float,float)
    };

    int main()  {
        Point p = Point(5.7, 1.2);   // Ambiguous: Which coordinate system?
    }

One way to solve this ambiguity is to use the Named Constructor Idiom:
    class Point {
    public: // These public static methods are so-called "named constructors"
        static Point rectangular(float x, float y);      // Rectangular coordinates     
        static Point polar(float radius, float angle);   // Polar coordinates        
    private:
        Point(float x, float y);                         // Rectangular coordinates
        float x_, y_;
    };

    inline Point::Point(float x, float y): x_(x), y_(y) { }
    inline Point Point::rectangular(float x, float y) { return Point(x, y); }
    inline Point Point::polar(float radius, float angle) { return Point(radius*std::cos(angle), radius*std::sin(angle)); }
Now the users of Point have a clear and unambiguous syntax for creating Points in either coordinate system:
    int main()  {
        Point p1 = Point::rectangular(5.7, 1.2);   // Obviously rectangular
        Point p2 = Point::polar(5.7, 1.2);         // Obviously polar
    }
2. How can I force objects of my class to always be created via 'new' rather than as 'locals' or 'global/static' objects ? 
Use the Named Constructor Idiom.  
    class Fred {
    public:
        // The create() methods are the "named constructors":
        static Fred* create()                 { return new Fred();     }
        static Fred* create(int i)            { return new Fred(i);    }
        static Fred* create(Fred const& fred) { return new Fred(fred); }
    private:
        // The constructors themselves are private or protected:
        Fred();
        Fred(int i);
        Fred(Fred const& fred);
    };

Now the only way to create Fred objects is via Fred::create():


    int main() {
        Fred* p = Fred::create(5);
        ...
        delete p;
    }
3. Composition and Initializer list: This is a method to call the argument constructor of a class from another class that has its objects by composition. 
    class base {
    public: 
        base()      { std::cout<<"base NO arg constructor"<<endl; } // This will not be called
        base(int a) { std::cout<<"base Single arg constructor"<<a<<endl; }
    };

    class derived {
    public:
        // base withArg(5);    // Error: Illegal to call be argumented constructor 
        base withArg;          // composition - This will not call No arg Constructor
        derived():withArg(5) { // This single arg initialization list will call the single arg constructor
        }
    };

    int main() {
        derived d; // base Single arg constructor 5
        return 0;
    }

4. What does the explicit keyword in C++ mean ? The explicit keyword makes a conversion constructor to non-conversion constructor. 
Any constructor that takes a single parameter and does not use the keyword "explicit" defines an implicit conversion (from the type 
of the  parameter, to the type of the object being constructed). It is usually good practice to declare single-argument constructors 
as explicit.

Also note that constructors with
	* default arguments for all parameters, or
	* default arguments for the second parameter onwards
can both be used as single-argument constructors. So you may want to make these also explicit.
    class Foo {
    public:
        Foo(int x) : m_x(x) { }
    private:
        int m_x;
    };

Objects of class Foo can be created in 2 ways:

    Foo bar1(10);
    Foo bar2 = 20; // conversion constructor

Depending upon the implementation, the second manner of instantiating class Foo may be confusing, or not what the programmer intended. 
Prefixing the explicit keyword to the constructor would generate a compiler error at Foo bar2 = 20;

        Init();
        // do something with nValue
    }
 
    void Init() { }
};
6. Is it legal (and moral) for a member function to say delete this?
As long as you're careful, it's OK for an object to commit suicide (delete this).
Here's how I define "careful": 

You must be absolutely 100% positively sure that
    - 'this' object allocated via new (not by new[], nor a local object on the stack, nor a global, nor a member of another object; but by plain ordinary new).
    - your member function will be the last member function invoked on this object.
    - the rest of your member function (after the delete this line) doesn't touch any piece of 'this' object (including calling any other member functions or
       touching any data members).
    - no one even touches the 'this' pointer itself after the delete this line. In other words, you must not examine it, compare it with another pointer,
       compare it with NULL, print it, cast it.

    void IMyInterface::Release() {
        --instanceCount;
        if(instanceCount == 0) delete this;
    }

7. After p = new Fred[n], how does the compiler know there are n objects to be destructed during delete[] p?
1. Uses an associative array with p as the key and n as the value. The run-time system stores the number of objects, n, somewhere where it can be
    retrieved if you only know the pointer, p.

2. When you allocate memory on the heap, your allocator will keep track of how much memory you have allocated. This is usually stored in a "head"
    segment just before the memory that you get allocated. That way when it's time to free the memory, the de-allocator knows exactly how much memory
    to free.

8. Why can't one make an object of abstract class? Give answer by compiler view ?
If there is a virtual function in your class the compiler automatically creates a table called virtual function table .. to store the virtual function addresses.... if the function is a pure virtual function then the vtable entry for that function will be NULL. Even if there is a single NULL entry in the function table the compiler does not allow to create an object.
